📝 PRD – Todo List ساده (Backend – [ASP.NET](http://ASP.NET) + SQL Server)  
1. 🎯 هدف محصول (Product Goal)  
  
هدف، توسعه یک API ساده برای مدیریت Todo List است که به کاربران اجازه می‌دهد پس از ثبت‌نام و ورود، کارهای روزانه خود را ایجاد، مشاهده، تکمیل و حذف کنند.  
  
2. 🧱 محدوده (Scope)  
در محدوده:  
احراز هویت (Register / Login)  
مدیریت تسک‌ها (CRUD ساده)  
ارتباط با دیتابیس SQL Server  
خارج از محدوده:  
رابط کاربری (Frontend)  
نوتیفیکیشن  
دسته‌بندی یا اولویت‌بندی تسک‌ها  
قابلیت‌های پیشرفته  
3. 👤 کاربران (Users)  
کاربران ثبت‌نام‌شده که می‌خواهند تسک‌های شخصی خود را مدیریت کنند  
4. 🔐 احراز هویت (Authentication)  
توضیح:  
  
کاربران باید بتوانند ثبت‌نام و ورود انجام دهند و برای دسترسی به APIها احراز هویت شوند.  
  
نیازمندی‌ها:  
ثبت‌نام با نام کاربری و رمز عبور  
ورود و دریافت JWT Token  
استفاده از Token برای دسترسی به APIها  
جلوگیری از دسترسی بدون لاگین  
5. 🧩 قابلیت‌های اصلی (Core Features)  
5.1 ایجاد تسک  
کاربر بتواند یک تسک جدید ثبت کند  
فقط شامل عنوان (Title)  
عنوان نباید خالی باشد  
5.2 مشاهده تسک‌ها  
کاربر بتواند لیست تسک‌های خود را مشاهده کند  
فقط تسک‌های همان کاربر نمایش داده شود  
5.3 تغییر وضعیت تسک  
کاربر بتواند وضعیت تسک را تغییر دهد:  
انجام شده (Completed)  
انجام نشده  
5.4 حذف تسک  
کاربر بتواند تسک را حذف کند  
فقط اجازه حذف تسک‌های خودش را دارد  
6. 🧠 User Flow  
کاربر ثبت‌نام می‌کند  
وارد سیستم می‌شود (Login)  
Token دریافت می‌کند  
درخواست‌های Todo را با Token ارسال می‌کند  
تسک اضافه می‌کند  
تسک‌ها را مشاهده می‌کند  
وضعیت تسک‌ها را تغییر می‌دهد  
در صورت نیاز تسک را حذف می‌کند  
7. 🗂️ مدل داده (Data Model)  
User  
Id  
Username (Unique)  
PasswordHash  
Todo  
Id  
UserId (FK)  
Title  
IsCompleted  
CreatedAt  
8. 🧾 قوانین کسب‌وکار (Business Rules)  
هر کاربر فقط به داده‌های خودش دسترسی دارد  
بدون احراز هویت، هیچ API Todo قابل دسترسی نیست  
Title تسک نمی‌تواند خالی باشد  
Username باید یکتا باشد  
حذف و تغییر فقط روی تسک‌های همان کاربر انجام شود  
9. ⚙️ نیازمندی‌های فنی (Technical Requirements)  
[ASP.NET](http://ASP.NET) Core Web API  
Entity Framework Core  
SQL Server  
JWT برای احراز هویت  
10. 🧪 معیار پذیرش (Acceptance Criteria)  
 کاربر بتواند ثبت‌نام کند  
 کاربر بتواند لاگین کند و Token دریافت کند  
 بدون Token دسترسی به APIها ممکن نباشد  
 کاربر بتواند تسک ایجاد کند  
 کاربر بتواند لیست تسک‌های خود را ببیند  
 کاربر بتواند وضعیت تسک را تغییر دهد  
 کاربر بتواند تسک را حذف کند  
 داده‌ها در SQL Server ذخیره شوند  
 هر کاربر فقط داده‌های خودش را ببیند  
11. 🎯 تعریف موفقیت پروژه  
APIها بدون خطای اصلی کار کنند  
احراز هویت به درستی اعمال شود  
عملیات CRUD کامل و صحیح انجام شود  
پروژه برای نمایش در رزومه یا تمرین مصاحبه قابل استفاده باشد
